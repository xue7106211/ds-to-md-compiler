## 一、概述

浮窗是大屏设备上的**模态面板**，以大屏上适当宽度的容器承载小屏上全屏页面的内容，旨在提升大屏的交互与视觉体验。

- **用途**：浮窗用于**非主线使用场景**（如设置、新建、查看次要信息），让用户在不离开主线的情境下，注意力聚焦在一个「临时」的次级任务中。因此，**浮窗不能是应用的第一级**，且应避免在主线场景使用浮窗。
- **形态**：MIUIX 浮窗组件随窗口/设备条件不同发生形态变化（具体见「四、响应式变化规则 - 形态」）：
  1. 居中大窗口状态
  2. 小屏全屏页面状态
- **本文约定**：除特殊提及外，「浮窗」均指**大屏下居中显示的浮出窗口**。

## 二、场景

### 1. 应用内打开设置

在系统应用内打开，用户对该应用进行设置，并即时生效。与「设置 app」内「设置 - 应用设置 - 系统应用设置」为同一页面。浮窗内多为列表。

### 2. 创建与编辑项目

用户创建、编辑某条目，浮窗内多为表单。

### 3. 其他

其他为优化大屏表现的场景，呈现次要信息、承载次要操作；场景较为丰富，业务根据需要自行设计。

### 常见使用场景

| 场景 | 说明 |
| --- | --- |
| **常见场景 1** | 文件夹、群组等项目的查看与管理。对象多为一条条目（如笔记项、联系人项）的群组。对非主线或非必须的项目，业务常用浮窗进行管理。 |
| **常见场景 2** | 放大页面上原有卡片，承接展示更多信息与操作。 |

## 三、结构

浮窗通常由以下部分构成：

### 1. 标题区域

浮窗的标题为 **MIUIX 标题栏组件**，需自行调用。Xiaomi HyperOS 标题栏组件的规范地址后期补充；此处仅写浮窗标题的建议，折行规则、副标题规则等由标题栏组件规范制订、详述。

| 要素 | 必选/可选 | 规则 |
| --- | --- | --- |
| **左侧导航** | 必选 | 浮窗是临时任务，需提供固定、清晰的关闭方式。浮窗内部首页，左侧 icon 为「取消」；若为小屏全页面形态，左侧 icon 为「返回」。浮窗内部二级页面及多级，左侧 icon 固定为「返回」。形态相关说明见本文「形态」章节。 |
| **标题文字** | 可选 | 浮窗不要求一定有标题文字；若浮窗仅有一级且足够明确、简单，则无需标题。 |
| **右侧 icon** | 可选 | 业务根据场景自行配置是否有右侧 icon 及功能。icon 规范地址后期补充。Figma 上需补充，线上有但当前 library 上没有。 |
| **边距** | — | 浮窗的标题栏距浮窗顶部固定边距 `8dp`。图见边距配图 🔗。 |

### 2. 内容区域

浮窗作为内容容器，内部区域开放给业务根据需要自行搭建。可选元素包括但不限于：表单/列表、插图、按钮、索引条、分割线等。此处仅规范标准边距与需特殊注意的元素。

| 规则 | 内容 |
| --- | --- |
| **边距** | 可参考边距规范。左右视觉边距为 `28dp`。若内容部分为图片/列表形式，则第一项距离标题栏 `12dp`。 |
| **按钮** | 可参考按钮/全屏页面的规范。若浮窗内容区有按钮，保证在浮窗第一屏内可见；考虑内容滑动时将按钮置底保留，置底时建议距底边距 `28dp`。 |
| **滑动条/索引条** | 可参考全屏页面规范。滑动范围在内容区，除自定义底部置底的区域之外。 |

### 3. 外部压黑

浮窗底层全屏压黑（`#000000`，`30%`），使用户注意力聚焦在当前任务上，并保留一部分上下文信息的呈现。业务可对点击交互进行定义，详见本文「五、交互规则」。

```yaml
visual_data:
  component: Floating Window
  overlay:
    color: "#000000"
    alpha: "30%"
  box_model:
    title_top_margin_dp: 8
    content_horizontal_margin_dp: 28
    content_first_item_to_title_dp: 12
    button_bottom_margin_dp: 28
```

## 四、响应式变化规则

### 1. 形态

| 设备 | 全屏 | 分屏 | 小窗 |
| --- | --- | --- | --- |
| Pad | 横全屏，竖全屏 | 横 2/3 分屏 | 4:3 的小窗 |
| 折叠屏内屏 | 横全屏，竖全屏 | / | / |

- 其他状态下，MIUIX 浮窗组件自动变换为**全屏页面形态**，可看做普通的移动端全屏页面。
- 形态改变时，标题栏导航、布局边距应发生对应变化，保证正确导航与合适视觉效果。见规范结构部分。
- 以 Pad K81 为例，浮窗在不同环境下的形态变化见源 PDF 图示。

### 2. 尺寸

居中窗口尺寸规则同时适用于 Pad 和折叠屏设备。（对于 4:3 小窗的窗口形态，根据小窗规则进行等比例缩小。）

| 维度 | 规则 |
| --- | --- |
| **浮窗宽度** | 固定为 `546dp`。 |
| **浮窗高度** | 横屏下为当前屏幕高度的 `80%`；竖屏下固定为 `636dp`。 |

```yaml
box_model:
  width_dp: 546
  height:
    landscape: "80%"
    portrait_dp: 636
```

### 3. 位置

- 左右居中。
- 上下居中；不减去顶部状态栏、底部虚拟按钮、小白条区域。

### 4. 键盘弹起

- 浮窗容器在键盘弹起时位置不变；内部内容（多为文本框组件）的上滑参见文本框组件规范「位置」。

## 五、交互规则

### 1. 关闭操作

浮窗可通过以下 4 种方式关闭：

| 操作 | 实现 | 行动 |
| --- | --- | --- |
| 1. 点击标题栏 X 按钮 / 全屏一级下的 ← 按钮 | 由业务实现 | 一般为直接关闭或起弹窗二次确认；也可配置其他行动。 |
| 2. 点击标题栏对号按钮（如果有） | 由业务实现 | 一般为直接关闭；也可配置其他行动。 |
| 3. 点击外部压黑 | MIUIX 提供 | 默认关闭浮窗。业务可拦截并配置其他行动，比如起弹窗二次确认。 |
| 4. 下拉标题栏顶部区域（下拉高度需超过浮窗高度一半，否则浮窗回弹至原位置） | MIUIX 提供 | 默认关闭浮窗。业务可停用此方式，不能拦截配置其他行动。 |

- 注意分场景考虑关闭操作的配置组合，以及给用户适当的反馈；详见本文「六、注意事项 - 4」。
- 除上述 4 种操作外，若用户只需做一个选择，业务可自定义**单击选项后关闭浮窗（并生效）**；详见注意事项 - 4。

**不推荐**：目前部分业务存在「全面屏上滑手势退出应用回到桌面，再返回应用后关闭浮窗」的方式，会直接丢弃用户已输入的信息和操作状态，建议业务自查并修改。

### 2. 层级跳转

- 浮窗窗口在当前应用的上层，从属于当前应用。浮窗内部的二级与多级在浮窗内**平行滑动切换**，动画部分详见动画规范（to-do）。
- 浮窗下级可通过以下 2 种方式返回上一级：
  - 点击标题栏 ← 按钮（二级及多级页面）
  - 系统导航的返回操作（全面屏手势/虚拟键），在浮窗内部逐级返回上一级
- **建议**：理论上不应存在浮窗过多跳转全屏的情况，以保证体验连续性。若系统中出现较多浮窗跳全屏，多为新页面适配大屏时未改为浮窗组件造成，建议业务自查并修改，保证同一应用内无浮窗跳全屏。

### 3. 浮窗内的滑动

若内容自定义区高度不满足业务需求，**可滑动范围为内容区不包含底部按钮的高度范围**（如有），不包括标题区。

## 六、注意事项

### 1. 谨慎选择浮窗场景

- 浮窗有离开主要进程、「临时态」的隐喻，用于非主线使用场景（如设置、新建、查看信息），应避免在主线重要场景使用浮窗。**浮窗不能是应用的第一级**。
- MIUIX 浮窗组件不支持位置和大小的改变，可能遮挡过多背景、不能做到位置近手近呼出，因此不能很好支持需要底层背景信息透出的业务场景。此类情况可重新考虑大屏架构，使用分栏、全屏、近手弹窗或其他架构。

### 2. 浮窗内层级应尽量扁平

- 浮窗适合次级、临时任务，不建议用浮窗承载过于复杂的任务与表单。浮窗内容尽量在 **2 级、2 屏以内**；若任务复杂可考虑全屏承载。

### 3. 尽量避免浮窗起弹窗

- 浮窗上再起弹窗的打扰性较强。除非是必要的二次确认，考虑将选项弹窗、近手弹窗等嵌入一级浮窗内。参考弹窗规范的注意事项 🔗。

### 4. 关闭操作需给出适当的反馈

- 浮窗有临时性、「转瞬即逝」感，因此应特别注意关闭/保存在用户心理层面的确认。

| 维度 | 建议 |
| --- | --- |
| **关闭的按钮** | 若用户只需做一个选择，做完就关闭浮窗（并生效），无需添加右上角或底部的按钮来确认；若用户需做多个选择，需要更强调而确定的关闭按钮，以「确认」「保存」「完成」。 |
| **关闭后直观反馈** | 关闭后，考虑返回浮窗底部页面时，用户是否能直观感受到新建/修改成功（如明显的底层新增卡片、定位到新增卡片、底层新增的动画、明显的 toast 等）。 |
| **点压黑关闭自动保存** | 因点击外部压黑较容易误触，新建/编辑任务场景下若设定点击外部浮窗关闭，考虑保存/创建用户已填写的内容。 |
| **关闭的策略** | 根据用户是否填写表单/编辑表单决定关闭策略。例如：若已更改/已填写，可配置「点击外部压黑不关闭浮窗」；若未填写，右上角 ✅ 置灰，视觉上提示此时不可新建。 |
